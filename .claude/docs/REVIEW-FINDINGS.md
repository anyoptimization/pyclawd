## [HIGH] Lint cannot enforce the docstring convention it claims — `convention="google"` doesn't reject NumPy sections, and the `DOC` rules are silently disabled
- **dimension:** docstring-convention  ·  **category:** Doc-vs-code drift / lint configuration
- **files:** pyproject.toml, scaffold/templates/pyproject.toml.tmpl, scaffold/templates/AGENTS.md.tmpl, AGENTS.md, SKILL.md
- **problem:** Running `ruff check src/pyclawd/project.py src/pyclawd/doctor.py` passes with `All checks passed!` despite both files being full of NumPy-style docstrings — empirical proof that the stated enforcement does not exist. Two distinct config gaps cause this:
1. ruff's `[tool.ruff.lint.pydocstyle] convention = "google"` (pyproject.toml line 76) only toggles which `D` rules are active; it does NOT validate that a docstring uses Google sections instead of NumPy sections. A well-formed NumPy docstring satisfies the always-on `D` rules, so it passes silently. The claim in AGENTS.md ("NumPy-style ... will fail lint") and the scaffold AGENTS.md.tmpl (line 100, same wording) is therefore false for the core failure mode.
2. The same ruff invocation emits `warning: Selection 'DOC' has no effect because preview is not enabled.` The `DOC` (pydoclint) family is listed in `select` (pyproject.toml line 71 and scaffold template line 52) but ruff only runs DOC rules under `preview = true`, which is set nowhere in this repo or in the scaffold template. So the `DOC` rules that AGENTS.md and SKILL.md (line 293: "`DOC` — parameter names in docstring match the actual signature") advertise as enforced are completely inert. Both the dogfooded config and every freshly scaffolded project inherit this dead selection.
Net effect: the docstring convention is documentation-only; nothing mechanical prevents drift, which is exactly how project.py and doctor.py slipped in.
- **fix:** Pick one of two coherent paths and align config + docs + dogfood:

Path A — make it real: add `preview = true` under `[tool.ruff.lint]` in pyproject.toml AND scaffold/templates/pyproject.toml.tmpl (activates the DOC/pydoclint rules and removes the warning), then convert the NumPy docstrings in src/pyclawd/project.py and src/pyclawd/doctor.py to Google style (Args:/Returns:, no types) so the dogfood is clean. Note ruff's `convention` still won't reject a well-formed NumPy `Parameters` block by itself, so if rejecting that exact pattern matters, add a dedicated scan (e.g. a `pyclawd ls`-style grep for `Parameters\n----------`) or pydoclint as a real step.

Path B — make docs honest: drop `DOC` from `select` in both pyproject.toml:71 and the scaffold template (kills the perpetual warning), and soften the absolute claims — change AGENTS.md:104, scaffold AGENTS.md.tmpl:86, and SKILL.md:250/285 from "NumPy-style fails lint / DOC enforced" to "Google style is the convention checked by reviewers/agents; ruff `D` checks docstring style but does not reject NumPy sections."

Either way, fix the line-number references in the finding when filing, and start by converting project.py/doctor.py so the canonical example matches whatever the docs end up claiming.

## [MEDIUM] load_project() raises TypeError/ImportError that escape every CLI boundary (degrade-never-crash gap)
- **dimension:** core-python  ·  **category:** error-handling consistency
- **files:** discovery.py:181-200, run.py:57-74, cli.py:84-95, doctor.py:294-302
- **problem:** `load_project()` wraps any failure inside `spec.loader.exec_module` into a clean `ConfigError`, but the post-import validation does NOT: it raises a bare `ImportError` when the spec/loader is None (line 182-183) and a bare `TypeError` when the config module has no module-level `project` attribute or it is the wrong type (lines 190-200). All CLI boundaries only catch `ConfigError` (`load_project_or_exit` run.py:65, `root` cli.py:89, `run_doctor` doctor.py:299, `dump_json` doctor.py:344, config.py:28). So a `.pyclawd/config.py` that imports cleanly but forgets `project = Project(...)` (or assigns the wrong type) produces a raw uncaught traceback and a generic exit 1 instead of the documented clean `✗` one-liner + exit 2. This is the most likely real-world misconfiguration and it violates both the "commands degrade, never crash" rule and the exit-code contract (2 = not-configured).
- **fix:** Make the missing/wrong-type `project` case and the None-spec case travel the same clean path as import failures. Simplest single-point fix: in discovery.py raise `ConfigError` (not TypeError/ImportError) at lines 182-183, 192-195, and 197-200, keeping the existing messages — then all five existing `except ConfigError` boundaries already emit `✗ ...` + exit 2 with no further changes. Update the `load_project` docstring `Raises:` section accordingly (drop TypeError/ImportError). Add a unit test that points the loader at a tmp_path `.pyclawd/config.py` lacking a `project` symbol (and one assigning a non-Project value) and asserts `ConfigError` is raised / the CLI exits 2.

If a soft no-op exit is truly desired for build, instead amend the exit-code contract in .claude/CLAUDE.md and AGENTS.md to call out build as the documented exception — but aligning to 2 is simpler and consistent.

## [MEDIUM] `test` uses manual verb dispatch + hand-rolled flag parsing, inconsistent with the `docs` sub-Typer
- **dimension:** commands-python  ·  **category:** design / consistency
- **files:** commands/test.py, tests.py
- **problem:** `docs` is implemented as a proper Typer sub-app with real subcommands and Typer-native options (e.g. docs.py docs_timings uses `top: int = typer.Option(...)`). `test` instead is a single passthrough command (commands/test.py:29-68) that manually inspects `ctx.args[0]` against a hardcoded verb set `{run,fast,all,failures,timings,fix}` (line 49), then forwards to tests.dispatch (tests.py:305-346), which AGAIN hand-parses `--top`/`--top=` and `--slow-threshold`/`--slow-threshold=` (lines 318-343). - **fix:** Apply three minimal fixes rather than a full sub-app rewrite (the passthrough is needed for arbitrary pytest args/categories). 1) Fix the exit-code collision in tests.py:331,342 — return a non-reserved code (e.g. typer's BadParameter / exit 1) instead of 2, so 2 stays "not configured". 2) Surface `--slow-threshold` in the `test` docstring (test.py:30-45) and in the `timings` help line so it is discoverable. 3) De-duplicate the verb list: define it once (e.g. a module-level constant of verbs) and reference it both in the test.py guard (line 49) and the dispatch routing, or fold the guard into dispatch returning a sentinel for unknown verbs. Optionally align the `--top` default between docs_timings (0=all) and test timings (25) for consistency.
